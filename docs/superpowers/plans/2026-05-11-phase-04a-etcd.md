# Phase 04a Etcd Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add default etcd-based service registration and service discovery for all runtime service-to-service calls.

**Architecture:** Add a thin `pkg/cloudwego` glue package that converts shared registry/discovery config into Kitex and Hertz options. Services own only config wiring and option assembly; no service imports another service's internal packages. Registry and discovery are required by default; static address service discovery is not a supported startup path.

**Tech Stack:** Go 1.25.6, CloudWeGo Kitex, CloudWeGo Hertz, `github.com/kitex-contrib/registry-etcd`, `github.com/hertz-contrib/registry/etcd`, Viper/mapstructure config.

- [ ] **Step 3: Implement config structs and option builders**

- `RegistryConfig`

- `KitexRegistryOptions`
- `KitexClientOptions`
- `HertzServerOptions`

Behavior:
- Disabled registry returns no options.
- Disabled discovery returns `client.WithHostPorts(staticAddr)`.
- Enabled config requires non-empty endpoints.
- Registry config requires non-empty service name.
- Hertz registration requires either `cfg.Addr` or `listenAddr`.
- Kitex registry/resolver uses prefix `<prefix-or-default>/kitex`.
- Hertz server registration uses `registry.Info` with `utils.NewNetAddr("tcp", addr)`.
