一页结论

AssetService 是生成资产平台的资产主数据服务。它把“资产类型定义”“用户资产实例”“版本快照”“媒体对象”“上传会话”分成相互独立但可组合的模型，edge-api 只负责 HTTP 适配，真正业务规则收敛在 asset Kitex 服务内部。

服务边界

edge-api 不访问资产 Mongo；它从登录态取 user_id，组装 base.BaseReq，经 Kitex 调 AssetService。asset 服务独占资产库、版本、媒体对象与 OSS 会话逻辑。

租户模型

第一版是个人资产库：workspace_id = user_id，created_by 也来自同一个 ObjectID。客户端不能传 workspace，跨用户隔离靠所有 repository 查询都带 workspace_id。

版本模型

AssetType.PartSchemas 定义结构，Asset 保存当前版本指针，AssetVersion.Parts 保存一次完整快照。版本不可变，修改内容必须创建新版本或复制旧版本再覆盖 part。

媒体模型

StorageUploadSession 管直传会话，MediaObject 管对象存储元数据。AS-04 的实现已接入 OSS presign/head，且 coverMediaId 与 version parts 的 mediaIds 会做 workspace 内存在性校验。

请求链路流转

整体链路是 Client → Kong → edge-api/Hertz → Kitex RPC → asset/AssetImpl → biz → repository → Mongo/OSS。下图把普通 CRUD、版本写入和媒体上传三条路径合在同一张请求流转图里。

关键点：HTTP 层只做绑定、分页默认值、错误码到 HTTP 状态映射；RPC handler 负责 DTO 与 model 互转；biz 层负责 workspace 推导、引用校验、版本规则和存储流程编排。

资产库 CRUD

  1. 客户端访问 /api/v1/assets/*，先过 Auth 中间件。
  2. AssetHandler 绑定 JSON/query/path，构造 BaseReq.UserID。
  3. AssetImpl 提取 userID，调用 AssetTypeBiz、AssetCategoryBiz 或 AssetBiz。
  4. biz 校验 ObjectID、名称、引用关系，并调用 repository。
  5. Mongo repository 查询时始终附带 workspace_id，返回 DTO 给 edge-api。

版本快照

  1. 创建版本或复制版本进入 AssetVersionBiz。
  2. 先加载同 workspace 的 Asset 与 AssetType。
  3. 按当前 PartSchemas 校验 part key、required、value kind 和内容格式。
  4. 校验 mediaIds 属于同 workspace 后生成下一个版本号。
  5. 插入 asset_versions，再更新 assets.current_version。

媒体直传

  1. 创建 upload session 时校验 content type、size、TTL。
  2. 服务端生成 object key，写入 storage_upload_sessions。
  3. OSS adapter 返回短期 PUT presigned URL。
  4. 客户端直传 OSS 后调用 finalize。
  5. asset 执行 HeadObject，创建 MediaObject 并把 session 标为 finalized。

数据结构与组合关系

AssetService 的设计核心是把“类型定义”和“资产内容”解耦：类型只定义 parts 的形状，版本才保存某一刻的实际内容。媒体对象作为独立资源被 Asset 和 AssetVersion 引用，而不是内嵌在资产文档里。

核心集合

集合 职责 关键字段 主要索引/约束
asset_types 用户自定义资产类型，定义 part schema。 workspace_id, code, part_schemas[] unique workspace_id + code；按 updated_at 分页。
asset_categories 个人资产库分类树，和资产类型不是同一概念。 parent_id, sort_order, created_by workspace_id + parent_id + sort_order；删除前检查子分类和资产引用。
assets 具体资产实例，保存资产库状态与当前版本指针。 type_id, saved_to_library, category_id, current_version, cover_media_id, provenance 按 library/type/category 三类维度分页；删除是软删除。
asset_versions 资产内容不可变快照，保存完整 parts。 asset_id, version, parts, change_reason, provenance unique asset_id + version；列表按 version desc。
media_objects 对象存储媒体元数据，供封面和 parts 引用。 provider, bucket, object_key, cdn_url, variants[], source unique provider + bucket + object_key；workspace 内分页。
storage_upload_sessions 前端直传 OSS 的临时会话与 finalize 幂等状态。 status, expires_at, content_type, size, sha256, media_id unique provider + bucket + object_key；按状态与过期时间查询。

组合语义

AssetType 和 AssetPartSchema 是一对多组合：删除类型前要确认没有资产引用它；schema 可以更新，但不会改写旧版本。

Asset 持有 TypeID、可选 CategoryID、可选 CoverMediaID，并以 CurrentVersion 指向当前激活版本号。

AssetVersion 的 Parts 是 map[string]AssetPartValue，key 必须来自当前 AssetType.PartSchemas。这是完整快照，不是增量 diff。

AssetPartValue 支持 text、json、media、mixed。JSON 会校验合法 JSON 文本；media 和 mixed 会校验 ObjectID 与同 workspace 的 MediaObject 存在性。

StorageUploadSession finalization 后写回 media_id，重复 finalize 可幂等返回已有 MediaObject。

Provenance 是多个主模型共用的溯源值对象，可挂 workflow run、step run、generation job、prompt 和扩展字段。

层级关系

分层的价值在于隔离协议、业务规则和持久化细节。AssetService 的每一层职责相对清楚：edge-api 做协议适配，asset handler 做 RPC DTO 适配，biz 做规则，repository 做 Mongo 操作，storage adapter 做 OSS 客户端封装。

层 路径 职责 边界约束
REST 门面 services/edge-api/router.go, handler/asset.go 登录态、参数绑定、分页默认值、HTTP 状态映射、调用 Kitex client。 禁止直接访问 asset Mongo，禁止 import asset 内部 biz/dal。
RPC 契约 idl/asset/asset.thrift 定义 DTO、请求响应、枚举、AssetService 方法。 请求统一带 base.BaseReq，响应统一带 base.BaseResp。
RPC Handler services/asset/handler.go nil request 保护、userID 提取、DTO/model 互转、错误码包装。 不承载复杂规则；复杂校验下沉到 biz。
Biz services/asset/biz/*.go workspace 推导、引用校验、版本不可变规则、上传会话编排。 只依赖 repository/storage 接口，不直接写 collection。
DAL services/asset/dal/model, dal/mongo Mongo 文档模型、索引、软删除/硬删除/分页查询。 集合访问封装在 repository 中。
Storage services/asset/storage 对象存储抽象、Aliyun OSS presign/head、OTel client span。 不放入 pkg，直到多个服务确实复用。

设计取舍与注意点

主题 当前选择 收益 代价或后续点
个人 workspace workspace_id = Base.UserID 最小权限模型清晰，所有查询天然按用户隔离。 后续团队资产库需要把 workspace 抽象成组织或空间 ID，并兼容历史数据。
schema 可变 AssetType.PartSchemas 始终可替换。 第一版简单，用户能快速调整资产结构。 旧版本不保存 schema snapshot，读取时只能原样返回 parts；强审计场景需补 schema history。
版本完整快照 AssetVersion.Parts 保存完整内容。 读取简单、回滚/复制语义直观。 大对象或频繁修改时存储会膨胀；未来可另做 diff/压缩，不影响当前 API。
版本号生成 读取最大版本号 + 1，唯一索引兜底。 实现短、Mongo 约束明确。 并发创建时当前代码未做自动重试，冲突会返回业务错误；高并发写入可补 retry 或原子计数。
媒体直传 asset 只签发 URL 和 finalize，不代理文件流。 降低 edge/asset 带宽压力；对象 key 和元数据仍由服务端管控。 需要客户端正确完成 PUT；更强安全可后续接回调验签、内容安全、缩略图任务。
OSS 适配位置 services/asset/storage 私有包。 避免过早把厂商 SDK 抽进 pkg。 当 model、notification 等服务也需要对象存储时，再抽通用能力更稳。
