# 二审 Review 规范

定义"使用第二个 agent / 工具 / 模型对当前改动做收尾 review"时的通用契约。不限工具——可以是 Codex review、Claude review、人工 reviewer、CI 静态检查、其他模型；任何"非编写者执行的复审"都遵守本规范。

> 与 `quality.md §代码审查要点` 的关系：那里定义"看什么"（正确性 / 最小性 / 一致性 / 安全性），本文件定义"怎么对待 review 结果"。

## 何时触发

- 用户显式要求二审 / autoreview / 第二个模型把关
- 完成非平凡（T2 / T3）改动后、最终提交或交付前
- 修复后再次审查本地分支或 PR 分支

T0 / T1 改动可跳过；但凡有用户要求或有交付仪式，必须走完。

## 七条契约

| # | 契约 | 含义 |
|---|---|---|
| 1 | **review 输出仅供参考** | 任何二审产物都不能盲目套用；所有"建议"在被验证前都不是事实 |
| 2 | **每条 finding 必须落地验证** | 阅读对应代码路径与相邻文件，确认 finding 描述与现实一致 |
| 3 | **依赖外部行为时读源** | finding 涉及第三方库、协议、类型时，回到依赖的文档 / 源码 / 类型定义核对，不靠记忆 |
| 4 | **拒绝不切实际的建议** | 推测性风险、不可能的 edge case、为修小问题做大重构、过度复杂化的方案——直接驳回 |
| 5 | **修改在最小责任边界内进行** | 优先小补丁；除非明显改善 bug 类别，否则不发起 refactor |
| 6 | **跑到 review 无可操作 finding 为止** | review 触发改动后必须重跑相关测试 + 重跑 review；循环直到清零 |
| 7 | **驳回时只在必要处加一行注释** | 仅当该决策代表真实不变量或责任归属、未来 reviewer 可能再问时，才写一行行内注释；不写废话 |

## 与 DEVLOOP 的对接

`devloop.md` 的"验证 → 交付"两步内嵌入二审 review：

```
... → 验证（自测通过）→ 二审 review（按本规范）→ 交付
```

任何在二审中触发的代码改动，回到"验证"重跑相关测试，再回到"二审 review"重跑工具，直到本规范第 6 条满足。

## 是否推送

**不要**为了让远端工具能 review 而提前 push。仅当用户明确要求 push / ship / 更新 PR 时才推送。本地 dirty 改动 / 未推送的分支也能完成本规范要求的 review（多数 review 工具支持 `--uncommitted` 或本地 diff 模式）。

## 上下文效率

二审输出通常嘈杂。推荐用子代理（subagent）做一次过滤，只让上层看到：

- 接受的可操作 finding
- 驳回的 finding（每条一行驳回理由）
- 需要重跑的具体文件 / 测试

只有改动极小、或环境不支持子代理时才把原始 review 直接呈现给上层。

## 最终报告（交付时附）

每次 review 闭环后给一份简短报告，至少含：

- 使用的 review 命令（含工具名、目标范围）
- 跑过的测试 / 证据
- 各 finding 的接受 / 驳回结论与一句理由
- 最终干净的 review 命令输出，或保留的某条 finding 的驳回理由

## 自查清单

- [ ] 是否在 T2 / T3 或用户要求时触发了二审？
- [ ] 每条 finding 是否经过实际代码 / 文档核对？
- [ ] review 触发的改动是否重跑了相关测试 + 重跑了 review？
- [ ] 是否避免了为 review 而提前推送？
- [ ] 最终报告是否包含命令、证据、接受 / 驳回结论？
