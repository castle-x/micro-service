# DEVLOOP 状态机

每次任务的必经入口。不跳步，不省略。

> **提交与版本控制**：DEVLOOP 定义"做什么、按什么顺序"，VCS 规范（`vcs.md`）定义"怎么提交、原子提交流程"。两者互补——DEVLOOP 的"交付"步骤对应 VCS 的"需求生命周期"（AI 研发 → AI 自测 → 人类确认 → 原子提交）。

## 状态流转

```
意图识别 → 分级 → 分支执行 → 验证 → 交付
```

## 任务分级

| 级别 | 定义 | 典型场景 | 流程严格度 |
|------|------|----------|------------|
| T0 | 微操作 | 改变量名、调格式、修 typo | 直接执行，无需 plan |
| T1 | 小任务 | 单点 bug、小调整 | 确认意图→执行→验证 |
| T2 | 中任务 | 新功能、非平凡改动 | 确认意图→plan→执行→验证→交付 |
| T3 | 大任务 | 重构、跨模块、架构变更 | 确认意图→plan（人类审批）→分步执行→验证→交付 |

## 分支执行

### 新功能分支

```
调研 → plan → 实现 → 测试 → 文档
```

- 调研：确认技术方案，产出简要记录
- plan：列出步骤和验证点
- 实现：按 plan 逐步执行
- 测试：单元测试 + 手动验证（T2）或 E2E（T3）
- 文档：更新相关文档

### Bug 修复分支

```
复现 → 定位 → 修复 → 回归测试
```

- 复现：必须先写出复现步骤
- 定位：找到根因，记录位置
- 修复：最小变更，不扩大范围
- 回归测试：**必须**同步写测试防止回归

### 重构分支

```
确认范围 → 保证测试通过 → 逐步重构 → 验证
```

- 先确认重构边界，列出影响文件
- 确保现有测试全部通过作为基线
- 每步重构后跑测试
- 完成后全量验证

### 调研分支

```
产出文档 → 人类确认
```

- 调研结果必须形成文档
- 不经确认不得直接进入实现

## 验证标准

验证步骤必须使用**可判定的成功标准**，禁止模糊描述。

| 错误示范 | 正确示范 |
|----------|----------|
| "看起来对了" | "测试用例全部通过" |
| "功能正常" | "`<项目 check 命令>` 无错误" |
| "改好了" | "E2E 测试 `X` 通过" |
| "没问题" | "人类确认预期行为" |

### 按级别的验证要求

| 级别 | 验证要求 |
|------|----------|
| T0 | 改动可见即可 |
| T1 | 功能正常 + lint 无新增错误 |
| T2 | 单元测试通过 + 手动验证核心路径 |
| T3 | 全量测试 + E2E + 人类验收 |

## 交付物

| 级别 | 交付物 |
|------|--------|
| T0 | 代码变更 |
| T1 | 代码变更 + 验证结果 |
| T2 | 代码变更 + 测试 + 更新文档（如涉及） |
| T3 | 代码变更 + 测试 + 文档 + 变更说明 |

## 规则

1. **意图不清则停止**——不猜测，不假设
2. **分级错误则升级**——宁可 T2 当 T3，不可 T3 当 T1
3. **验证失败则回退**——回到执行步骤，不跳到交付
4. **每步只做一步**——不合并步骤，不并行有依赖的步骤
5. **人类是最终决策者**——T3 的 plan 和所有不确定的决策需人类确认
6. **三轮止猜**——同一个问题或需求，3 轮问答仍未满足用户预期时，禁止继续猜测试错，必须转而搜索互联网寻找答案
