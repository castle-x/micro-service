# 生成资产平台总体路线图

> 当前文档只记录已讨论出的总体方向。服务细节尚未全部完成设计，具体实现前应继续拆分到各服务 roadmap/spec。
>
> 最后更新：2026-05-13

## 背景与目标

平台目标不是做一个纯聊天式生图工具，而是让用户把生图经验沉淀为可复用的数字资产，并通过可配置的引导流程持续出图。

核心用户包括：

- 自运营虚拟 IP 账号，需要快速稳定地产出角色相关图片
- 写真、影楼等商业出图场景，需要复用风格、人物、套系和参考图
- 后续可能扩展到宠物、汽车、电商产品等任意可结构化描述的资产类型

核心价值：

- 用户可以自定义资产类型和资产组成部分
- 用户可以把提示词、图片、结构化数据沉淀到资产中
- 用户可以通过工作流逐步生成、确认和保存资产
- 用户可以复用资产组合生成新图片，并继续微调、编辑、沉淀

## 当前设计判断

- `Asset` 是平台的核心沉淀物，但第一版不做复杂资产状态机。
- 资产是否进入资产库只用 `savedToLibrary` 区分；未保存的结果属于历史产物。
- 用户先定义资产类型，再定义资产包含哪些部分。
- 每个资产部分可存储的内容类型由平台提供，例如文本、图片、结构化 JSON、混合内容。
- 工作流不是传统自动化引擎，而是串行分步的 Guided Interactive Process。
- 每个节点可以多轮对话、多次生图、多次调整，最终把结果写入某个资产部分，或保留为历史产物。
- 复用资产生图也属于工作流能力：消费已有资产，产出新的历史产物或资产内容。

## 初步服务边界

| 服务 | 当前定位 | 设计状态 |
|---|---|---|
| `asset` | 管理资产类型、资产实例、资产版本、资产部分、媒体资源、OSS/CDN 对接和生产溯源 | 已基本敲定，优先实现 |
| `workflow-service` | 管理工作流模板、运行实例、步骤状态、节点产出和写入资产的规则 | 方向明确，细节待继续讨论 |
| `prompt-service` | 管理提示词结构、抽取、模板、编译和上下文组装 | 方向明确，暂不先实现完整能力 |
| `generation-service` | 管理批量生成任务、模型调用任务、重试、结果回写和扣费前置 | 方向明确，需结合 model/credits 细化 |
| `model-service` | 继续作为模型网关，适配外部 LLM / 图像模型供应商 | 已有服务基础，后续增强 |
| `credits/billing` | 负责额度、扣费、订单和资金相关流程 | 后续阶段接入 |

## 用户交互主线

### 创建资产类型

用户定义一种资产类型，例如“角色资产”。

资产类型包含若干部分，例如：

```text
角色资产
├─ 背景
├─ DNA
├─ 脸
├─ 身体
└─ 风格
```

每个部分声明允许保存的内容类型：

```text
背景：文本 + 结构化 JSON
DNA：结构化 JSON
脸：图片 + 文本 + 结构化 JSON
身体：图片 + 文本 + 结构化 JSON
风格：文本 + 图片
```

### 创建资产

用户基于资产类型创建一个资产实例，例如“角色 A”。

工作流逐步引导用户完成：

```text
背景讨论 → DNA 结构化 → 脸部生图 → 身体生图 → 风格确认 → 保存到资产库
```

每一步产物可以：

- 写入当前资产的某个部分
- 作为历史产物保留
- 被用户保存到资产库

### 复用资产生图

用户选择已有资产和提示词/风格/参考图，启动生产型工作流：

```text
选择资产 → 选择目标 → 组合上下文 → 批量生图 → 选择结果 → 微调/编辑 → 保存或保留历史
```

## 阶段路线图

| Phase | 主题 | 状态 | 产物 |
|---|---|---|---|
| 01 | `asset` 基础能力 | 待启动 | 资产类型、资产实例、资产版本、资产部分、媒体对象、OSS/CDN 基础能力；见 [`../asset/roadmap.md`](../asset/roadmap.md) |
| 02 | `workflow-service` 基础能力 | 未拆 spec | 工作流模板、运行实例、步骤状态、节点输出和写入资产部分 |
| 03 | 提示词抽取与编译能力 | 未拆 spec | 按用户定义结构抽取提示词、保存模板、编译生图上下文 |
| 04 | 生图任务与模型调用链路 | 未拆 spec | 批量生成、重试、回调、结果写入 asset |
| 05 | 资产复用生产流 | 未拆 spec | 消费已有资产组合出图，支持选择、微调、保存 |
| 06 | 额度/计费与商业化能力 | 未拆 spec | credits/billing 接入，按模型调用和资源消耗扣费 |

## 当前未确认问题

- 工作流第一版允许用户自定义到什么程度：只编辑模板，还是允许新增/删除节点。
- `prompt-service` 是否第一期独立成服务，还是先作为 workflow/generation 内部模块。
- `generation-service` 与现有 `model-service` 的边界需要在实现前再明确。
- CDN 策略是第一期只做配置与 URL 抽象，还是直接接入真实 CDN 刷新/签名能力。
- 历史产物的保留策略、清理策略和额度计费策略尚未确定。

## 下一步

优先进入 `asset` 实现准备：先完成资产类型、资产实例、资产版本、资产部分、媒体对象、OSS 抽象和基础 CRUD，再回到工作流如何写入资产部分的设计。
